#+RESULTS:

This is a guide that explains how to run the epimight pipeline to estimates genetic
correlation between two disorders stratified by birth year.

- Any data shown in this guide comes from [[../data/pipeline-tte.csv][this randomly generated time-to-event dataset]]
  - 10 000 individuals
  - 2 disorders (schizophrenia and coronary artery disease)
  - 2 types of relatives (parent/offspring and full siblings)
- All code shown in this guide have been written to [[./guide.R][this R file]]

The basic concept of the pipeline is pretty simple, you create a dataset that contains
time-to-event for all disorders and types of relatives you want to analyse. Then you
tell the pipeline which pair of disorders and type of relatives you want to analyse and
it will output cumulative incidences, heritabilities and genetic correlations.

The most work you have to do is to format the data in a way that the pipeline expects.

The diagram below gives you an overview of how the pipeline works.

* Setting up the pipeline

To setup the pipeline, we start by importing the epimight library:

#+BEGIN_SRC R :exports code :eval no
library(epimight)
#+END_SRC

And then create a new instance with our TTE pool (the variable ~tte~ is the data table shown in the previous chapter):

#+BEGIN_SRC R :exports code :eval no
pipeline <- Pipeline$new(pool = tte)
#+END_SRC

#+RESULTS:

The supplied pool must be a ~data.table~ with the following required columns:

- ~person_id~
- ~disorder~
- ~failure_status~
- ~failure_time~
- ~relationship_kind~
- ~relatives~
- ~relatives_diagnosed~

It accepts any additional columns used for stratification.

* Running trials

After creating the pipeline instance, you run trials by supplying the ~run_trial~ function with:

- Disorder 1
  - Identifier
  - Earliest onset
  - Latest onset
- Disorder 2
  - Identifier
  - Earliest onset
  - Latest onset
- Relationship kind
- How many draws to perform

For example, if you want to run a trial between schizophrenia and coronary artery disease of full siblings,
using 2 draws, you would run:

#+BEGIN_SRC R :exports code :eval no
trial1 <- pipeline$run_trial(
  disorder1 = list(
    id             = "SCZ",
    earliest_onset = 1,
    latest_onset   = 100
  ),
  disorder2 = list(
    id             = "CAD",
    earliest_onset = 1,
    latest_onset   = 100
  ),
  relationship_kind = "FS",
  draws = 2,
  group_columns = list("born_at_year", "gender")
)
#+END_SRC

#+RESULTS:

** Using trial results

The ~run_trail~ function returns a list with the following members:

- ~args~ :: The arguments the trail was run with
- ~errors~ :: Any draw errors encountered
- ~results~ :: Each independant draw result
- ~meta~ :: Draw results meta analyzed accross stratification within draws
- ~rubin~ :: Draw results rubin combined across draws within stratification
- ~rubin_meta~ :: Rubin combines meta analyzed

*** args

The member ~args~ is just a copy of the arguments supplied to the ~run_trial~ function.
It's output for debugging and reproducibility purposes.

*** errors

The member ~errors~ is list a unnamed list of each draw that had an error. It's output for debugging purposes.

*** results

The member ~results~ is a ~data.table~ containing resulting rows of each draw performed, with the column
~draw~ added, that contain the index of the draw that produced the rows.

Each row contains all statistical values produced by the analyses run. Each analysis has it's columns
prefixed after this naming convention:

 - ~re_d1_c1~ = Risk estimates of disorder 1, cohort 1
 - ~re_d1_c2~ = Risk estimates of disorder 1, cohort 2
 - ~re_d1_c3~ = Risk estimates of disorder 1, cohort 3
 - ~re_d2_c1~ = Risk estimates of disorder 2, cohort 1
 - ~re_d2_c3~ = Risk estimates of disorder 2, cohort 3
 - ~h2_d1~ = Heritability of disorder 1 (cohort 1 vs cohort 2)
 - ~h2_d2~ = Heritability of disorder 2 (cohort 1 vs cohort 3)
 - ~gc_d1_d2~ = Genetic correlation (disorder 1 vs disorder 2)

* How it works :noexport:

You have a set of disorder pairs you want to produce genetic correlations between and time-to-event data
of a sample population you want to produce it from.

Each time-to-event entity represents:

1. A distinct individual
2. A distinct disorder
3. A distinct type of relatives
4. The disorder failure status/time of the individual
5. The amount of relatives the individual have
6. The amount of relatives diagnosed with the disorder the individual have

Here are 3 examples of what that could look like for the disorder schizophrenia and for the relatives type "full siblings":

- Individual 897
  - Diagnosed with schizophrenia at age 19
  - Has 2 full siblings
  - 1 of the siblings also has a schizophrenia diagnosis
- Individual 345
  - No schizophrenia diagnosis, was alive at end of study
  - Has 3 full siblings
  - None of the siblings has a schizophrenia diagnosis
- Individual 193
  - No schizophrenia diagnosis, died at age 30
  - Has no full siblings
  - None of the siblings has a schizophrenia diagnosis

You collect this kind of time-to-event data for all disorders and types of relatives you
want to produce genetic correlations for and feed it into the pipeline. This is called
the pipelines time-to-event "pool".

Then for each pair of disorders and type of relatives you want to produce a genetic
correlation for, you run a "trial". Here are 3 examples of what that could look like:

- ADHD vs. MDD in full siblings
- ADHD vs. ASD in parent/offspring
- ADHD vs. Bipolar type 2 in first cousins

By having a pool of all time-to-event data and have the trials select the specific data,
you only have to prepare your time-to-event once. Adapting the time-to-event data before
each trial is handled by the pipeline.

** The process

As mentioned before, a trial produces genetic correlations for a distinct pair of disorders and a type
of relatives. It does this in multiple interdependant steps. First it pulls out the time-to-event data for
the selected pair of disorders and type of relatives from its time-to-event pool.

It then separates the time-to-event data into 3 cohorts:

- ~c1~ = Target cohort
- ~c2~ = Individuals with at least 1 diagnosed relative of disorder 1
- ~c3~ = Individuals with at least 1 diagnosed relative of disorder 2

It then produces the following risk estimates (cumulative incidence):

- ~re_d1_c1~ = Risk estimates of disorder 1, cohort 1
- ~re_d2_c1~ = Risk estimates of disorder 2, cohort 1

It then copies over the time-to-event cohorts and the first risk estimates to a separate
analysis process (called a "draw") that is repeated ~K~ amount of times.

It then aggregates all draw results and returns:

- The arguments the trial was run with
- Each independant draw result
- Any draw errors encountered
- The aggregated draw results

The sub-chapters below describes how draws are performed, how draw results are aggregated and
how downsampling of relatives are performed.

*** Performing draws

Each draw performs the following steps:

- [[#downsampling][Downsample the diagnosed relatives]] of cohort ~c2~
- [[#downsampling][Downsample the diagnosed relatives]] of cohort ~c3~
- Produce risk estimates
  - ~re_d1_c2~ = Risk estimates of disorder 1, cohort 2
  - ~re_d1_c3~ = Risk estimates of disorder 1, cohort 3
  - ~re_d2_c3~ = Risk estimates of disorder 2, cohort 3
- Produce heritabilities
  - ~h2_d1~ = Heritability of disorder 1 (~re_d1_c1~ vs ~re_d1_c2~)
  - ~h2_d2~ = Heritability of disorder 2 (~re_d2_c1~ vs ~re_d2_c3~)
- Produce genetic correlation ~gc_d1_d2~ from risk estimates and heritabilities
- Return all results joined together as a single dataset
  - ~re_d1_c1~
  - ~re_d1_c3~
  - ~re_d2_c1~
  - ~h2_d1~
  - ~h2_d2~
  - ~gc_d1_d2~

*** Aggregating draw results

  :CUSTOM_ID: meta_analysis
  :END:

- Meta analyze across stratification within draws
- Rubin combine across draws, within stratification
- Meta analyze rubin combine across all
